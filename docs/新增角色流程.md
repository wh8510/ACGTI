# ACGTI 新增角色流程

本文档用于说明在当前仓库中新增一个角色时，需要改哪些文件、按什么顺序改、改完后如何自查。

## 1. 先理解角色在项目里的落点

当前项目里，一个“角色”并不是只存在于一个文件中，而是由下面几部分共同组成：

- `src/data/characters.json`
  角色的核心资料，包含角色代码、标题、匹配 MBTI、原型、标签、解析文案、六维向量等。
- `src/data/characterVisuals.json`
  角色视觉配置，包含结果页实际使用的图片路径和主题色。
- `public/images/characters/`
  角色立绘图片资源。角色必须有图，且应为透明背景素材。
- `src/data/characterProbabilities.json`
  角色命中概率配置，用于结果页展示“人群命中率”。
- `src/i18n/characters.ts`
  角色名和作品名的多语言映射表，用于国际化展示。

运行时的关系如下：

1. `quizEngine.ts` 根据答题结果对 `characters.json` 中的角色进行排序。
2. `characterVisuals.ts` 用 `characterVisuals.json` 中的图片和主题色补全角色展示信息。
3. `characterProbability.ts` 从 `characterProbabilities.json` 读取该角色的命中率。
4. `src/i18n/characters.ts` 为角色名和作品名提供 `zh-CN`、`zh-TW`、`en`、`ja` 的展示文本。

所以，新增角色至少要保证“资料、视觉、素材、概率、i18n”五处能对上同一个 `id`。

同时，本项目对新增角色还有两个硬性要求：

- 角色必须配套图片，且图片必须处理成透明图层。
- 每次新增角色后，都必须基于最新角色库全局重跑一次稀有度（命中概率）数据，不能只手动补一个值。

## 2. 新增角色的标准流程

推荐按下面顺序操作，避免中途出现引用断裂。

### 步骤 1：先确定角色基础方案

在落文件前，先把下面这些内容定下来：

- `id`
  全仓库唯一，建议使用英文小写加连字符，例如 `kaname-madoka`。
- `name`
  角色展示名。
- `series`
  作品名。
- `code`
  对外展示的角色自定义代码，这是结果页核心标签。
- `matchCode`
  角色的底层 MBTI 映射，例如 `INFP`。
- `matchCodeFlex`
  可选。若角色允许一定弹性匹配，可补充若干兼容 MBTI。
- `archetypeId`
  必须是现有 8 个原型之一。
- `vector`
  六维向量：`expression`、`temperature`、`judgement`、`order`、`agency`、`aura`。
- `tags`、`title`、`note`
  结果页会直接展示，文案风格要符合当前项目的“严肃测试外壳 + 二次元角色解析”方向。

其中 `archetypeId` 目前只能从以下值里选：

- `luminous-lead`
- `icebound-observer`
- `oathbound-captain`
- `trickster-orbit`
- `gentle-healer`
- `shadow-strategist`
- `chaos-spark`
- `moonlit-guardian`

### 步骤 2：在 `characters.json` 新增角色主体

编辑 `src/data/characters.json`，新增一个对象。建议直接参考现有角色的结构，不要随意发明新字段。

最小结构如下：

```json
{
  "id": "new-character-id",
  "name": "角色名",
  "series": "作品名",
  "matchCode": "INFP",
  "code": "ABCD",
  "title": "结果页称号",
  "archetypeId": "moonlit-guardian",
  "tags": ["标签1", "标签2", "标签3", "标签4"],
  "note": "结果页角色解析文案。",
  "vector": {
    "expression": 0,
    "temperature": 2,
    "judgement": 1,
    "order": 0,
    "agency": 1,
    "aura": 2
  }
}
```

可选增强字段：

- `image`
  可写，但当前项目最终展示优先取 `characterVisuals.json`，所以这里不是主入口。
- `matchCodeFlex`
  用于允许一个角色兼容多个 MBTI。
- `signature.uniqueAxes`
  用于强调该角色在六维向量上的独特判别点。
- `signature.questionAffinity`
  用于把某些题目答案倾向额外绑定到该角色，提高辨识度。

`questionAffinity` 的格式示例：

```json
{
  "signature": {
    "questionAffinity": [
      { "questionId": "q12", "expected": "agree", "weight": 1.4 },
      { "questionId": "q20", "expected": "neutral", "weight": 1.2 }
    ]
  }
}
```

注意事项：

- `id` 必须唯一，并且后续所有文件都用同一个值。
- `matchCode` 必须符合 MBTI 四字母格式。
- `vector` 六个轴必须写全。
- `code` 不要和已有角色重复。
- `questionId` 必须能对应到现有题库中的题号，例如 `q1` 到当前题库最后一题。

### 步骤 3：补角色视觉配置

编辑 `src/data/characterVisuals.json`，为同一个 `id` 增加配置：

```json
"new-character-id": {
  "image": "/images/characters/角色图片.png",
  "accent": "#7ec7f9"
}
```

这里的 `image` 和 `accent` 会被 `src/utils/characterVisuals.ts` 注水到最终结果对象中。

注意事项：

- `image` 必须指向 `public/` 下可以访问到的资源路径。
- `accent` 建议选择和角色视觉一致的主色，结果页顶部色块、按钮和强调区会用到。

### 步骤 4：放入角色图片资源

把角色图片放到：

`public/images/characters/`

建议遵循当前仓库现状：

- 文件名尽量与角色中文名一致，便于人工管理。
- 必须使用透明背景的人物立绘。
- 构图尽量适配结果页头部大图展示，避免头部被裁切过多。

如果需要重新处理素材，仓库中已有相关脚本可参考：

- `scripts/remove_character_backgrounds.py`
- `scripts/process_character_assets.py`
- `scripts/generate_character_webp.py`

这里的要求不是“有图就行”，而是：

- 新角色必须有图片，不能留空。
- 图片必须先完成抠图，确保主体是透明图层，而不是白底或带杂色背景的整张图。
- 放进 `public/images/characters/` 的最终资源，必须和 `characterVisuals.json` 的路径完全对应。

如果原图不是透明背景，新增角色前应先完成抠图处理，再接入项目。

### 步骤 5：补充角色命中概率

新增角色后，不应只在 `src/data/characterProbabilities.json` 里手填一个值，而是要基于最新全量角色库重新生成整份稀有度数据。

当前仓库里已有模拟脚本：

- `scripts/probability-simulation.mjs`

标准要求：

- 每次新增角色，都必须全局重跑一次稀有度。
- 重跑后，整体回写 `src/data/characterProbabilities.json`。
- 不建议只追加一条 `"new-character-id": x`，因为这会让整体分布失真。

最终 `src/data/characterProbabilities.json` 中的 `probabilities` 会是类似下面的结果：

```json
"new-character-id": 1.25
```

这里的值用于结果页展示“该角色在人群中的命中比例”，运行时由 `src/utils/characterProbability.ts` 读取。

注意事项：

- 所有角色最好都有值，否则前端会回退为 `0`。
- 这是展示用数据，不会直接参与角色排序，但会影响结果页可信度和完整性。
- 稀有度必须以“全量角色一起重算”为准，不能把新增角色当成局部补丁处理。

### 步骤 6：补齐 i18n 映射

编辑 `src/i18n/characters.ts`，至少补两类映射：

- `characterNameI18n`
  为新角色的 `id` 增加多语言名称。
- `seriesI18n`
  如果新角色来自一个之前未收录的新作品，也要为该作品补多语言名称。

角色名示例：

```ts
'new-character-id': {
  'zh-CN': '角色简中名',
  'zh-TW': '角色繁中名',
  en: 'Character English Name',
  ja: 'キャラクター名',
}
```

作品名示例：

```ts
'作品名': {
  'zh-CN': '作品名',
  'zh-TW': '作品繁中名',
  en: 'Series English Name',
  ja: '作品日本語名',
}
```

注意事项：

- 这一步不是可选项，新增角色默认需要做 i18n 适配。
- 如果只在 `characters.json` 里写中文名，而不补 `src/i18n/characters.ts`，英文、日文、繁中环境下会退回原始值，展示会不完整。
- 若角色所属作品是已有作品，只需复用已有 `seriesI18n` 项；若是新作品，必须新增该作品的多语言映射。
- `characterNameI18n` 的 key 必须与角色 `id` 完全一致。

## 3. 新增角色时的判定建议

为了让新增角色在结果上“像这个角色”，而不是只挂一个名字，建议按下面逻辑补数据。

### 3.1 `matchCode` 怎么定

- 先定底层 MBTI。
- 这个值决定角色在大方向上的命中范围。
- 如果角色存在边界模糊区，可以加 `matchCodeFlex` 做弹性兜底。

### 3.2 `archetypeId` 怎么定

- `matchCode` 决定底层人格型。
- `archetypeId` 决定角色更接近哪一种二次元原型表现。
- 两者最好保持一致，不要出现明显互相冲突的设定。

可以直接参考 `src/utils/quizEngine.ts` 中 `TYPE_TO_ARCHETYPE` 的映射关系。

### 3.3 `vector` 怎么定

`vector` 是角色细分排序的重要依据。当前六个轴为：

- `expression`：表达外放程度
- `temperature`：情感温度
- `judgement`：判断/决断倾向
- `order`：秩序与规划倾向
- `agency`：主动性与推动力
- `aura`：存在感与气场

建议做法：

- 先找 1 到 2 个仓库里最接近的已有角色作参照。
- 不要一上来把所有值都拉满。
- 真正有辨识度的地方，再通过 `signature.uniqueAxes` 或 `questionAffinity` 强化。

### 3.4 `note` 和 `tags` 怎么写

虽然项目是二次元包装，但结果页文案不能只剩梗图。

建议保持这个平衡：

- `title` 负责记忆点和角色称号。
- `tags` 负责快速识别特征。
- `note` 负责把“人格倾向 + 角色观感”串起来。

不要把角色描述写成医学、诊断或心理治疗口吻。

## 4. 改完后的自查清单

至少检查下面这些点：

- `src/data/characters.json` 中新增角色的 JSON 结构合法。
- `id` 在所有文件中完全一致。
- `src/data/characterVisuals.json` 已补对应配置。
- `public/images/characters/` 中确实存在对应图片。
- 图片已经完成抠图，背景为透明图层。
- `src/data/characterProbabilities.json` 已补该角色概率。
- `src/data/characterProbabilities.json` 是基于全量角色重新生成后的结果，而不是手工局部追加。
- `src/i18n/characters.ts` 已补角色名的多语言映射。
- 如果角色来自新作品，`src/i18n/characters.ts` 也已补作品名的多语言映射。
- `matchCode` 为合法 MBTI。
- `archetypeId` 为合法原型 ID。
- `vector` 六轴齐全。
- `questionAffinity` 引用的题号真实存在。

## 5. 建议的验证流程

新增角色后，建议至少做一次最小验证：

```bash
npm run build
```

然后人工检查：

1. 启动本地站点，进入角色图鉴页或结果页。
2. 确认新增角色图片能正常显示，没有 404。
3. 确认图片已经是透明图层，没有白底、实底或明显抠图毛边。
4. 确认结果页主色、标题、标签、解析文案显示正常。
5. 切换至少一种非简中语言，确认角色名和作品名没有漏翻或退回异常值。
6. 如有调试入口或定向测试方式，检查该角色是否能被正常命中。

如果后续要做更稳的内容扩充，建议再补两类验证：

- 用固定 MBTI 样本检查该角色是否落在预期区间。
- 检查全局重跑后的稀有度分布是否合理，避免角色库扩充后命中率失真。

## 6. 一次典型新增的实际改动清单

一次完整的新增角色，通常会改这几处：

- `src/data/characters.json`
- `src/data/characterVisuals.json`
- `src/data/characterProbabilities.json`
- `src/i18n/characters.ts`
- `public/images/characters/角色图片.png`

如果还涉及素材处理或概率重算，可能还会临时使用：

- `scripts/process_character_assets.py`
- `scripts/remove_character_backgrounds.py`
- `scripts/probability-simulation.mjs`

## 7. 最后原则

新增角色时，优先保证以下三件事：

- 角色代码对外展示明确。
- 底层 MBTI 与原型映射自洽。
- 文案、视觉、概率、素材四部分保持一致。

此外，下面两条应视为默认强约束：

- 没有透明抠图图片的角色，不算完整接入。
- 没有在新增后全局重跑稀有度的角色，也不算完整接入。
- 没有完成 i18n 适配的角色，也不算完整接入。

如果只补了名字和图片，没有补齐向量、原型、概率和结果文案，这个角色在当前项目里就还不算完整接入。
